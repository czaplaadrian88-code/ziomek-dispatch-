# 03 — ANALIZA LUK: stan obecny vs standard branżowy

> Wielki Audyt Integracji — FAZA 3A. Data: 2026-07-05.
> Metoda: `00-STAN-OBECNY.md` (checklist + 4 audyty kodu) × `02-BENCHMARK.md` (rozdział „Standard branżowy": MUST 1–10, SHOULD 11–17).
> Nakład: **S** ≤ 2 osobodni · **M** ≤ 2 osobotygodnie · **L** ≤ 6 osobotygodni · **XL** > 6 osobotygodni. Priorytet: **MUST** (bez tego integracje nie ruszą) / **SHOULD** (przewaga, szybko po starcie) / **COULD** (później).
> Typy partnerów z FAZY 1 (kogo luka blokuje): **HUB** (Restimo, Deliverect), **GIEŁDA** (Restaumatic, UpMenu), **POS** (GoPOS, Dotykačka, Poster…), **E-COM** (WooCommerce/Shopify/Presta), **AGREGATOR-SELF** (self-delivery Pyszne/Bolt), **DIY** (restauracja z własnym devem).

## Pakiet „INTEGRATION READINESS v1"

Minimalny, spójny zakres — wyprowadzony z luk MUST. Bez KAŻDEGO z tych punktów pierwsza zewnętrzna integracja nie ruszy albo skompromituje się w pierwszym tygodniu. Kolejność = zależności.

| Krok | Zakres (luki) | Nakład |
|---|---|---|
| **IR-0. Warunki wstępne bezpieczeństwa** | L14 (host-firewall, bindy lokalne) + przegląd sekretów połączeń | S |
| **IR-1. Higiena powierzchni** | L13 (prefiks `/v1`, rate-limit na kluczu, CORS) + L8 (model błędów + katalog kodów) + L9 (jedna idempotencja `external_delivery_id`) | M |
| **IR-2. Publiczne API DaaS v1 (tor jedzenia + paczki)** | L3 (`POST /v1/quotes` z TTL) + L2 (`POST /v1/deliveries`) + L6 (`GET`) + L7 (`cancel` z polityką) + L15 (`delivery-areas`) + L16 (COD) + L5 (tracking_url w odpowiedziach) — wszystko spięte z ISTNIEJĄCĄ ścieżką OPS-02/`ingest_inbound_order`, nie nowy silnik | L |
| **IR-3. Kanon zdarzeń** | L4 (most silnik→StatusEvent w czasie rzeczywistym; maszyna przejść jedzenia; anulacja/8/9) | L |
| **IR-4. Worker webhooków** | L1 (HTTP push, podpis Stripe-style, backoff, wersjonowanie; katalog 10 statusów kanonu) | M |
| **IR-5. Sandbox + docs + onboarding** | L10 (test-tenant + symulator) + L11 (developer portal) + L12 (self-service connections + panel UI) | M/L |
| **IR-6. Formalności** | L24 (wzorzec DPA/RODO) | S |

**Suma nakładu IR v1: ~2× L + 3× M + 3× S ≈ 10–14 osobotygodni inżynierskich** (jedna osoba full-time ~3 miesiące; dwie ~6–7 tygodni). SHOULD (L17–L21) wchodzą zaraz po v1 — z czego konektor Deliverect (L20) i GPS-push (L17) jako pierwsze, bo dają przewagę sprzedażową.

**Co świadomie POZA v1:** inwersja źródła prawdy / obejście gastro (L21 etap 2 — XL, osobny program), OAuth2 (L23), zwroty (L22), POD-foto (L18 etap 2 — wymaga sprintu apki).

## Zależności i ryzyka pakietu

1. **IR-3 (kanon zdarzeń) jest sercem** — webhooki (IR-4) bez niego będą kłamać (panel maruderem). Kolejność: IR-3 przed publicznym włączeniem IR-4, choć budowa równoległa.
2. **Reużycie zamiast budowy:** IR-2 spina się z istniejącym `create_delivery`+`_push_to_ziomek` (ścieżka formularza restauracji) — NIE tworzymy drugiego toru zleceń. Idempotencja mapuje się na istniejący `idempotency_key`.
3. **Flip flag AUT-06/08 = zmiana live** — wg protokołu zmian Ziomka (ACK, testy bazowe, rollback), poza peakiem.
4. **Gastro w środku ścieżki pozostaje pojedynczym punktem awarii w v1** — świadoma decyzja; partnerom komunikować SLA realistycznie; program L21-etap-2 planowany w architekturze docelowej (04) tak, by API publiczne NIE wymagało zmian przy przejściu.
