## 3. Root cause = time_matrix bez DWELL (HYPOTHESIS A CONFIRMED)

### Code path proof

**`route_simulator_v2.py:759-769`:**
```python
time_matrix: List[List[float]] = [[0.0] * N for _ in range(N)]
for i in range(N):
    for j in range(N):
        if i == j:
            continue
        try:
            time_matrix[i][j] = max(0.0, float(leg_min(i, j)))  # ONLY travel time
        except Exception:
            time_matrix[i][j] = 9999.0
```

**`route_simulator_v2.py:_simulate_sequence` (post-process, line 416-509):**
```python
t = t + timedelta(minutes=leg_min(current, idx))
# ...
if node["kind"] == "pickup":
    t = t + timedelta(minutes=DWELL_PICKUP_MIN)  # +2 min per pickup
# ...
elif node["kind"] == "drop":
    t = t + timedelta(minutes=DWELL_DROPOFF_MIN)  # +2 min per drop
```

**Asymetria:** TSP solver dostaje `time_matrix = travel_only`, post-process dolicza `DWELL × 2N stops`. Solver myśli "ok in window", post-solve assertion (`V3.27.6 FIX 2b`) wykrywa walked > close + 0.5 → reject + greedy fallback.

### Quantitative model fits empirics

| Bag | Stops added | DWELL accum | Excess vs window (10 min) | Predicted reject | Empirical reject |
|---|---|---|---|---|---|
| 1 | 2 (1p + 1d) | +4 min | 4-10 → marginal | low | 3.4% ✓ |
| 2 | 4 | +8 min | 8-10 → borderline | medium | 34.0% ✓ |
| 3 | 6 | +12 min | 12 > 10 → fails | high | 57.9% ✓ |
| 4 | 8 | +16 min | well over | very high | 86.0% ✓ |
| 5 | 10 | +20 min | guaranteed | 100% | 100.0% ✓ |

Model fits empiryczny pattern w **lockstep** — strong evidence że time_matrix bez DWELL jest dominującym mechanism mass-reject.

## 4. Logger routing observability gap

### Empirical finding
V3274_OR_TOOLS_VIOLATION + V3274_TIMEWINDOW_FALLBACK warnings są **TYLKO w `czasowka.log`** (214 entries today). Brak w `dispatch.log`, `shadow.log`, `panel_watcher.log`, `shadow_dispatcher.log`.

### Root cause
`route_simulator_v2._log = logging.getLogger("route_simulator_v2")` — bez explicit handler.
Default propagation=True → events propagate do **root logger**.

W procesie czasowka_scheduler (oneshot timer) — root logger MA handler (likely setup_logger("czasowka_scheduler", "czasowka.log") side-effect lub propagation chain).
W procesie shadow_dispatcher (long-running) — root logger NIE ma handler → events lost.

### Konsekwencje
1. Mass-reject events z **shadow path** (dispatch_pipeline w shadow_dispatcher) — invisible
2. Adrian widzi tylko czasówki path violations w czasowka.log
3. LGBM training audit trail incomplete

### Fix recommendation (FAZA 1 collateral)
Add explicit handler dla route_simulator_v2 logger:
```python
# common.py setup_logger ALE w shadow_dispatcher entry point + dispatch_pipeline:
_route_log = setup_logger("route_simulator_v2", "/root/.openclaw/workspace/scripts/logs/route_simulator.log")
```

Lub dodać handler na root w shadow_dispatcher startup.

## 5. Rekomendacja FAZA 3 ścieżka

### Empirical evidence

- DWELL accumulation per bag size matches reject rate growth

### REKOMENDACJA: ŚCIEŻKA A — time_matrix correction

Implementation sketch:
```python
# route_simulator_v2.py:_ortools_plan, line 759-769 replacement
time_matrix: List[List[float]] = [[0.0] * N for _ in range(N)]
for i in range(N):
    for j in range(N):
        if i == j:
            continue
        try:
            travel = max(0.0, float(leg_min(i, j)))
        except Exception:
            travel = 9999.0
        # V3.28: include DWELL of ARRIVING node (j) in cumulative time
        # — reflects post-process _simulate_sequence which adds DWELL after arrival
        kind_j = nodes[j].get("kind")
        if kind_j == "pickup":
            travel += DWELL_PICKUP_MIN
        elif kind_j == "drop":
            travel += DWELL_DROPOFF_MIN
        time_matrix[i][j] = travel
```

### Predicted impact (if FAZA 3 ścieżka A deployed)
- Bag=1 reject: 3.4% → **~0%** (already low)
- Bag=2 reject: 34.0% → **~5-10%** (DWELL math now in solver)
- Bag=3 reject: 57.9% → **~15-25%** (residual = OSRM jitter ±2-3 min)
- Bag=4 reject: 86.0% → **~30-40%** (window ±5 prawdopodobnie nadal too tight dla 4-bag chains, may need ścieżka B residual calibration)
- Bag=5 reject: 100% → **~50-60%** (4-stop deep chain with travel variance)

### Residual considerations
- Bag>=4 nawet po DWELL fix może wymagać dodatkowo dynamic window (ścieżka B residual)
- Bag>=4 stops = inherent >20 min span, even perfect path nie fits ck ±5 dla wszystkich orderów
- Hard truth: **±5 min commit window dla 4-bag chain z różnymi ck = mathematically infeasible** dla part orderów (gdy ck spaced <2-3 min apart)

### Recommended additional in FAZA 3
- Po wdrożeniu ścieżki A — **re-audit reject rate distribution**
- Jeśli bag>=4 reject rate >40% post-fix → ścieżka B residual: window relax dla bag>=3 stops do ±(5 + (bag_size-1)) min
- ALBO ścieżka C (KOORD escalation) dla bag>=4 z spread ck
