## 1. Problem (jednolinijkowy)

OR-Tools solver dostaje `time_matrix[i][j] = leg_min(travel)` BEZ DWELL. Post-process `_simulate_sequence` dolicza `DWELL_PICKUP=2 + DWELL_DROPOFF=2` per stop. Solver myśli "ok in window", post-solve assertion (`V3.27.6 FIX 2b`) wykrywa walked > close+0.5 → reject + greedy fallback (bez window awareness).

Math: bag=N → +4N min skumulowane DWELL vs window=10 min wide. Bag=2 (8 min DWELL accumulated) borderline; bag=3 (12 min) zawsze przekracza; bag=5 (20 min) 100% reject.

## 3. Implementation sketch

### 3.1 Helper function (new, `route_simulator_v2.py` przed `_ortools_plan`)

```python
def _dwell_min_for_arriving(node: dict) -> float:
    """V3.28 ścieżka A: DWELL component of time_matrix[i][j].

    Service time consumed at arriving node (j). Used to align solver
    semantyka z _simulate_sequence pickup_at storage convention
    (post-DWELL = "leaves restaurant"). FAZA 0 audit empirycznie
    confirmed: bag>=2 reject rate 34-100% explained by DWELL accumulation
    not seen by solver. Symmetric DWELL_PICKUP_MIN = DWELL_DROPOFF_MIN = 2.0.
    """
    kind = node.get("kind")
    if kind == "pickup":
        return DWELL_PICKUP_MIN
    if kind == "delivery":
        return DWELL_DROPOFF_MIN
    return 0.0  # courier depot or unknown — no service time
```

### 3.2 Modified time_matrix loop (`route_simulator_v2.py:759-769`)

```python
# V3.28 ścieżka A (2026-05-XX): time_matrix[i][j] = travel + DWELL_at_j.
# Pre-fix: solver dostawał travel only, post-process dolicza DWELL → asymetria
# 4N min ukrytego time per N stops. Window check post-solve assertion fail.
# Math (bag=N stops): 2*N pickups+drops × DWELL=2min = 4N min pre-fix unseen,
# now visible to solver → respects [ck-5, ck+5] frozen window correctly.
time_matrix: List[List[float]] = [[0.0] * N for _ in range(N)]
for i in range(N):
    for j in range(N):
        if i == j:
            continue
        try:
            travel = max(0.0, float(leg_min(i, j)))
        except Exception:
            travel = 9999.0
        # V3.28 ścieżka A: include DWELL of arriving node (j) in cumul time.
        # Reflects post-process _simulate_sequence: pickup_at = arrival + DWELL.
        if _common.ENABLE_V328_TIME_MATRIX_DWELL:
            travel += _dwell_min_for_arriving(nodes[j])
        time_matrix[i][j] = travel
```

## 4. Edge cases analysis

### 4.1 Courier depot node (i=0, j=0)
- `nodes[0] = {"kind": "courier", ...}` (line 181)
- `_dwell_min_for_arriving({"kind": "courier"})` → 0.0 (return default)
- `time_matrix[0][j] = travel(courier, j) + DWELL_at_j` ✅
- Solver: cumul[0] = 0 (depot), cumul[j] = 0 + travel + DWELL = arrival + DWELL = post-service. ✅

### 4.2 Already-picked-up bag stops (delivery only)
- nodes added via line 277: `{"kind": "delivery", ...}` (NO pickup pair)
- `_dwell_min_for_arriving` → DWELL_DROPOFF_MIN ✅
- Pickup pre-occurred outside this route — kurier wiezie food, drop only.
- Window for these is [0, V327_DROP_TIME_WINDOW_MAX_MIN=120] (loose, line 843-845).
- DWELL inclusion still correct semantically (cumul = post-drop).

### 4.3 Super-pickup (V3.26 Fix 7 group_oids)
- Single pickup node represents N orders (line 256-273)
- `_simulate_sequence` line 452: `t += DWELL_PICKUP_MIN` ONCE for whole group
- Time matrix: DWELL_PICKUP at this node ONCE — matches. ✅
- Group_oids stored separately in pickup_at dict — display logic intact.

### 4.4 Drop nodes order_id=None (theoretical)
- `_dwell_min_for_arriving` checks `kind`, returns 0 for unknown kinds
- Defensive — never crash on malformed nodes

### 4.5 Wait at restaurant (pickup_ready_at)
- `_simulate_sequence:446-451`: if `t < ready_utc`, `t = ready_utc` (wait jump)
- Solver does NOT model wait — windows assume earliest=arrival
- Edge: solver picks order where arrival << ck-5; window check: cumul[j] = arrival + DWELL << ck → **REJECTS** (cumul < window_open)
- BUT V3.27.4 sets window_open = max(0, open_min - 5) — clamps to 0 if courier could arrive before ck-5
- After FAZA 3: window check applied post-DWELL. Arrival may be much earlier than ck (kurier zaplanowany blisko, czeka). Post-DWELL = arrival + 2 min still well before ck-5 if arrival ≪ ck-5.
- This is OK: solver doesn't reject "early arrival + wait" because cumul[j] respects window [max(0, open_min-5), open_min+5]. If DWELL+arrival < open_min-5 → wait happens in real, post-process advances pickup_at to post-wait+DWELL; window check uses pickup_at post-wait+DWELL = real time, in [open-5, open+5] (assuming real arrival ok).
- **Subtle:** w obecnym kodzie window check używa `walked = pickup_at - now` (post-wait + DWELL). Po FAZIE 3 solver widzi "post-DWELL bez wait" — może under-estimate cumul[j]. Ale post-solve assertion nadal sprawdza pickup_at post-wait+DWELL vs window. Jeśli pickup_ready << ck (kurier czeka), post-wait push push pickup_at do ~ck (jak kurier zaczyna pickup gdy ready). Then post-DWELL = ck + 2. Window [ck-5, ck+5] respects.
- TLDR: wait dynamics niezmienione, ścieżka A nie regresuje.

### 4.6 OSRM fallback / 9999.0 sentinel
- Gdy `leg_min` exception → `time_matrix[i][j] = 9999.0` (legacy)
- Po fix: `9999.0 + DWELL = 10001.0` — nadal sentinel, solver INFEASIBLE z 9999/10001 cost.
- Defensive: keep sentinel pattern intact.

### 4.7 Same-restaurant grouping (V3.26 Fix 7)
- Pre-grouping node count reduces N
- DWELL applied once per pickup node = once per group ✅
